S-01 Owner สมัครร้านใหม่ (First-time Setup)

เจ้าของร้านอาหารต้องการเปิดใช้ระบบ QRforPay ครั้งแรก
Precondition: ยังไม่มี account ในระบบ
1

เปิดหน้าเว็บ QRforPay

เห็นหน้า Landing พร้อม Hero, Features, Pricing และปุ่ม "ทดลองฟรี 7 วัน" + "สมัครเลย"

Route: / (app/page.tsx)
2

กดสมัคร → หน้า Register Step 1

กรอก ชื่อ-นามสกุล + อีเมล + รหัสผ่าน (8+ ตัว มีตัวเลข) กด "ถัดไป"

Validation: minLength 8, ต้องมีตัวเลข 1 ตัว
3

Register Step 2 — ข้อมูลร้านค้า

กรอก "ชื่อร้าน" + "PromptPay ID" (เบอร์โทร 10 หลัก หรือ เลขนิติบุคคล 13 หลัก)

Route: /register (app/register/page.tsx)
4

กด "สมัครสมาชิก"

supabase.auth.signUp() → self_register_shop RPC → สร้าง shop + profile ทันที

DB: INSERT shops + UPDATE profiles (role='owner', shop_id=ใหม่)
5

Redirect → เข้าระบบ POS ทันที

Redirect ไป /pos/tables (instant activation) หรือ /pending (ถ้า RPC ล้มเหลว)

S-02 Super Admin อนุมัติร้าน

Super Admin ตรวจสอบและอนุมัติร้านใหม่ที่สมัครเข้ามา
Precondition: มี pending owner อย่างน้อย 1 คน (จาก S-01)
1

Super Admin Login

Login → ระบบตรวจ role = 'super_admin' → Redirect ไป /pos/settings

2

เห็นรายชื่อ Pending Users

รายการ Owner ที่รอการอนุมัติ แสดง: email, ชื่อร้าน (pending), เลข PromptPay

3

กดอนุมัติ

approveOwner() → สร้าง row ใน shops table + อัพเดท profiles: role = 'owner', shop_id = ใหม่

4

Owner เข้าระบบได้ทันที

Owner refresh/re-login → profile.role = 'owner' → Redirect ไป /pos/sessions

S-03 Owner ตั้งค่าร้าน + เพิ่มพนักงาน

หลังร้านได้รับอนุมัติ Owner ตั้งค่าร้านให้พร้อมใช้งาน
Precondition: Owner login แล้ว, ร้านอนุมัติแล้ว (S-02 เสร็จ)
1

ไปแท็บ "ตั้งค่า"

เห็น: ชื่อร้าน, เลข PromptPay, จำนวนโต๊ะ (default: 10)

Route: /pos/settings
2

แก้ไขข้อมูลร้าน

แก้ชื่อร้าน / PromptPay ID / จำนวนโต๊ะ (1-200) → กด "บันทึก"

DB: UPDATE shops SET name, promptpay_id, table_count
3

เพิ่ม Cashier (พนักงาน)

กรอก: ชื่อ-นามสกุล, Email, Password → กด "สร้าง Cashier"

Action: Supabase Edge Function → auth.users + profiles (role='cashier')
4

ดูรายชื่อทีม

แสดงสมาชิกทั้งหมดในร้าน: email, ชื่อ, role — Owner สามารถลบสมาชิกได้

S-04 Owner จัดการสินค้า + เมนู

เพิ่ม/แก้ไข/ลบ/จัดลำดับสินค้าในร้าน
Precondition: Owner login, ร้านตั้งค่าแล้ว
1

ไปแท็บ "สินค้า"

แสดงรายการสินค้าทั้งหมด — ค้นหาได้ — สีบอกสถานะสต็อก (แดง=หมด, ส้ม=ใกล้หมด≤5, น้ำเงิน≤20)

Route: /pos/products
2

เพิ่มสินค้าใหม่

กดปุ่ม "+" → Modal form: ชื่อ, ราคา, หมวดหมู่, จำนวนสต็อก, รูปภาพ → บันทึก

DB: INSERT products
3

แก้ไข / ลบสินค้า

กดที่สินค้า → Modal เดิม (pre-filled) → แก้ข้อมูล → บันทึก | ลบ → Soft delete (is_active = false)

S-05 ลูกค้าสั่งอาหารเอง (Self-Order via QR)

Flow หลักของระบบ — ลูกค้านั่งโต๊ะ สแกน QR สั่งอาหารเอง ไม่ต้องเรียกพนักงาน
Precondition: ร้านมีสินค้าในระบบแล้ว + QR Code ติดบนโต๊ะ
1

สแกน QR Code บนโต๊ะ

ลูกค้าเปิดกล้องมือถือ → สแกน → เปิด browser อัตโนมัติ

URL: /order?session={customer_session_uuid}
2

โหลดหน้าเมนูร้าน

ระบบสร้าง customerSessionRef (UUID) → โหลด shop info + categories + products ผ่าน anon Supabase client

3

เลือกเมนู + เพิ่มลงตะกร้า

กดเลือกหมวดหมู่ (อาหาร, เครื่องดื่ม) → กด "+" เพิ่มจำนวน → ยอดรวมที่ด้านล่าง

State: localStorage cart (CART_KEY per session)
4

ตรวจสอบตะกร้า + กดยืนยันออเดอร์

เห็นรายการทั้งหมด + ราคารวม — แก้จำนวน ลบ หรือเพิ่มได้ → กด "ยืนยัน"

DB: INSERT orders (status=pending) + order_items + payments (status=pending)
5

ออเดอร์ส่งสำเร็จ → POS ได้รับ realtime

แสดงหน้า success → ตะกร้าถูกล้าง → POS ได้รับแจ้งเตือน realtime ทันที

Supabase Realtime broadcast → POS orders page

Screenshots — S-05 Customer Order

เปิดหน้า Order ไม่มี session
เปิดด้วย session ไม่ถูกต้อง
ต้องมี session UUID จริง (จาก QR Code บนโต๊ะ) ถึงจะเห็นเมนูร้าน — ถ้าไม่มีจะแสดง error ป้องกันการเข้าถึงโดยไม่สแกน QR

S-06 ลูกค้าจ่ายเงิน PromptPay QR

หลังสั่งอาหารแล้ว ลูกค้าจ่ายเงินผ่าน QR PromptPay ของร้าน
Precondition: ลูกค้าสั่งอาหารแล้ว (S-05 เสร็จ step 5)
1

กด "ชำระเงิน"

ลูกค้ากดปุ่มจ่ายเงินในหน้า customer

2

ระบบสร้าง PromptPay QR

EMV QR payload ตามมาตรฐาน BOT: Merchant ID (เบอร์/เลขภาษี) + จำนวนเงิน + CRC16 checksum

lib/qr.ts → generatePromptPayQR(promptpayId, amount)
3

แสดง QR Code + รอการชำระ

หน้าจอแสดง QR PromptPay + ยอดเงิน + สถานะ "รอการชำระเงิน..." + countdown 5 นาที

กรณี A: Auto Confirm

5a. Supabase Realtime ตรวจจับ payments.status = 'success' → auto completeOrder()

6a. หน้าจอเปลี่ยนเป็น ✓ สำเร็จ → ออเดอร์ badge: Auto

กรณี B: Manual Confirm

5b. แคชเชียร์กดยืนยัน → POS ฝั่งแคชเชียร์ กดปุ่ม "ยืนยันรับเงินแล้ว" → Dialog confirm

6b. ออเดอร์ badge: Manual + ชื่อ

Timeout: ถ้า 5 นาทีไม่มีการจ่าย → QR หมดอายุ → ลูกค้าเลือก "สร้าง QR ใหม่" หรือ "ยกเลิก"

S-07 ลูกค้าสั่งเพิ่ม (Multi-Order บนโต๊ะเดียว)

ลูกค้าสั่งอาหารรอบ 2, 3 บนโต๊ะเดิม — ระบบรองรับหลาย order ต่อโต๊ะ
Precondition: ลูกค้าสั่งไปแล้ว 1 ออเดอร์ (S-05 เสร็จ)
1

อยู่หน้าเมนูเดิม (session ยังอยู่)

ตะกร้าว่าง แต่ confirmedItems แสดงออเดอร์ก่อนหน้า + สถานะครัว (realtime)

2

เลือกเมนูเพิ่ม + ยืนยัน

กดเพิ่มรายการใหม่ลงตะกร้า → กดสั่ง → ระบบสร้าง order_id ใหม่ + order_number ใหม่ บนโต๊ะเดิม

DB: new row in orders (same table_number, same customer_session_id)
3

POS เห็นออเดอร์ใหม่ realtime

หน้าออเดอร์ POS มีรายการใหม่ดังขึ้น (highlight 8 วินาที + vibrate)

Bill Merging Rule: เมนูเดียวกันจากหลายออเดอร์ ต้องรวมเป็นบรรทัดเดียว เพิ่มจำนวน ห้ามซ้ำ (merge by product_id)

S-08 แคชเชียร์รับออเดอร์ + จัดการครัว

ออเดอร์เข้ามา → แคชเชียร์อัพเดทสถานะจนเสร็จ
Precondition: มีออเดอร์ pending เข้ามา (จาก S-05 หรือ S-09)
1

แจ้งเตือนออเดอร์ใหม่

Realtime subscription ตรวจจับ order ใหม่ → สั่น (Vibration) + Toast banner "ออเดอร์ #XX โต๊ะ Y"

2

ดูรายละเอียดออเดอร์

แท็บ "ออเดอร์" → Active tab → เห็น order card: เลขออเดอร์, โต๊ะ, รายการ, ยอดเงิน, สถานะ payment

Route: /pos/orders — Active Orders Tab
3

กด "กำลังทำ" → กด "พร้อมเสิร์ฟ"

pending → preparing → ready — ลูกค้าเห็น realtime

4

ยืนยัน Payment + Complete

QR: กด "ยืนยันรับเงินแล้ว" (manual) หรือ auto detect | เงินสด: กรอกจำนวนเงินรับ → คำนวณเงินทอน

completed — Order จบ

S-09 แคชเชียร์สั่งแทนลูกค้า (POS Order)

ลูกค้าบอกปากเปล่า → แคชเชียร์กดสั่งให้ผ่านหน้าจอ POS
Precondition: Cashier/Owner login แล้ว
1

ไปแท็บ "สินค้า" หรือ หน้า POS

เห็นรายการสินค้าทั้งหมด — กดเพิ่มสินค้าที่ลูกค้าต้องการ

2

ตรวจสอบตะกร้า

Cart screen: ดูรายการ + ราคา + ใส่ส่วนลด (% หรือ บาท) + เลือกเลขโต๊ะ

3

เลือกวิธีจ่าย

ทางเลือก A: QR PromptPay

กด "จ่ายด้วย QR" → แสดง QR ให้ลูกค้าสแกน → รอ confirm (auto/manual)

ทางเลือก B: เงินสด

กด "จ่ายเงินสด" → กรอก "เงินที่รับ" → ระบบคำนวณเงินทอน → ยืนยัน

ทางเลือก C: บันทึกไว้ก่อน (Save as Pending) — สร้าง order status=pending โดยยังไม่จ่ายเงิน → ครัวเริ่มทำได้เลย → จ่ายทีหลัง

S-10 แคชเชียร์ยกเลิกรายการ / ยกเลิกออเดอร์

2 ระดับ: ยกเลิกรายเมนู (item-level) หรือยกเลิกทั้งออเดอร์
Precondition: มีออเดอร์ในระบบ
กรณี A: ยกเลิกรายเมนู

1. กดดูรายละเอียดออเดอร์ → เปิด Detail Modal → เห็นทุก item

2. กดปุ่มยกเลิกข้าง item → confirm dialog

3. ระบบคำนวณยอดใหม่ (เฉพาะ active items)

4. ถ้าทุกรายการถูกยกเลิก → ยกเลิกทั้งออเดอร์อัตโนมัติ

DB: order_items.item_status='cancelled', item_cancelled_by, item_cancelled_at

กรณี B: ยกเลิกทั้งออเดอร์

1. เปิด Detail Modal ของออเดอร์

2. กด "ยกเลิกออเดอร์" → confirm dialog

3. ออเดอร์ย้ายไป History tab สถานะ cancelled

DB: orders.status='cancelled', cancelled_at, cancelled_by | payments.status='failed'

S-11 จัดการโต๊ะ + ย้ายโต๊ะ

ดูสถานะโต๊ะทั้งร้าน แชร์ QR ให้ลูกค้า ย้ายออเดอร์ไปโต๊ะอื่น
Precondition: Cashier/Owner login, มีโต๊ะตั้งค่าแล้ว
1

ไปแท็บ "โต๊ะ"

แสดง Grid ปุ่มเลขโต๊ะ 1..N — สีบอกสถานะ: 🟢 ว่าง 🟠 มีออเดอร์

Route: /pos/tables — สถานะ derive จาก active customer_sessions
2

กดเลือกโต๊ะ → ดูรายละเอียด

Detail Panel: ออเดอร์ทั้งหมดของโต๊ะ (combined view) + QR Code + ปุ่ม action

3

แชร์ QR / Copy URL

"แชร์ QR" → Screenshot + native Share API | "Copy URL" → clipboard

4

ย้ายโต๊ะ

กด "ย้ายโต๊ะ" → Modal เลือกโต๊ะปลายทาง → ยืนยัน → อัพเดท table_number ของทุก active order

DB: UPDATE orders SET table_number WHERE table_number=old AND status IN (pending, preparing, ready)
